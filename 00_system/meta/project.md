
# Project 运作规范（Project Playbook）

> 本文用于定义：
> **什么是 Project、什么时候建、如何更新、什么时候结束**

---

## 1️⃣ 什么是 Project（先划清边界）

### ✅ Project 是什么

一个 Project 必须同时满足：

* 有 **明确目标**
* 需要 **多天 / 多步骤推进**
* 会产生 **可复盘的结果**

典型例子：

* 一个长期维护的平台
* 一个较大的 feature
* 一次架构重构
* 一个需要周 / 月总结的工作任务

---

### ❌ Project 不是什么

以下内容 **不应该建 Project**：

* 单次排查问题
* 临时协助
* 一天内能完成的任务
* 纯学习 / 纯资料收集

👉 这些应进入：

* Daily（事实）
* Inbox（草稿）
* Notes（知识）

---

## 2️⃣ 什么时候创建 Project

### 推荐触发条件（满足一个即可）

* 这个事情 **今天做不完**
* 我需要 **反复回到这个事情**
* 我预计 **会为它写复盘**
* 它会占用我 **一段时间的注意力**

📌 如果你在犹豫：

> **“它会出现在我的 Daily 里 ≥ 3 次吗？”**
> 是 → 建 Project

---

## 3️⃣ Project 的最小结构

### 最小可用（MVP）

```
project_xxx/
└── index.md
```

### 推荐常态结构

```
project_xxx/
├── index.md      ← 唯一入口
├── tasks.md      ← 任务与进度
└── log.md        ← 关键过程
```

### 长期项目（可选）

```
project_xxx/
├── index.md
├── tasks.md
├── log.md
├── decisions.md
└── retro.md
```

📌 原则：

> **文件数 ≤ 5，否则说明拆分粒度有问题**

---

## 4️⃣ Project 内各文件的职责（防混乱）

### `index.md` — 项目控制台

* 项目目标
* 当前状态
* 当前关注点
* 指向其他文件

**不允许出现：**

* 过程日志
* 大段思考
* 每日记录

---

### `tasks.md` — 行动层

* 当前 Sprint
* Backlog
* 已完成

**规则：**

* Task 是“决定要不要做”
* 不记录当天做了什么（那是 Daily 的事）

---

### `log.md` — 变化记录

只记录：

* 新发现的问题
* 思路转变
* 风险暴露
* 关键进展

**不是：**

* 项目 Daily
* 工作流水账

---

### `decisions.md` — 关键选择（可选）

只在以下情况写：

* 当时存在分歧
* 未来可能被质疑
* 决策有代价

---

### `retro.md` — 回顾与调整（可选）

* 周 / 月 / 阶段性使用
* 不写新事实
* 只总结已有内容

---

## 5️⃣ Project 的日常运作方式（最重要）

### Daily 中如何引用 Project

Daily 里 **只允许一行**：

```md
- [[project_xxx]]：推进了 [[project_xxx/tasks#🔥 当前 Sprint]] 中的「xxx」
```

Daily：

* 不写任务列表
* 不写技术细节

---

### Task 什么时候更新？

只在三种情况下更新 `tasks.md`：

1. 任务完成
2. 任务拆分 / 延期 / 放弃
3. Sprint 切换

---

### Log 什么时候写？

满足任一条件就写：

* 发现了之前没想到的问题
* 推翻了原方案
* 识别出风险或技术债

---

## 6️⃣ Feature 与长期 Project 的关系

### 长期维护项目的推荐方式

* **一个长期 Project**
* 多个 feature 作为：

  * 子 Project
  * 或链接的 feature note

示例：

```md
## 🧩 Feature
- [[feature__rule_engine_refactor]]
- [[feature__baseline_optimization]]
```

📌 不建议：

* 每个小改动都新建 Project
* 也不建议所有 feature 都塞进一个文件

---

## 7️⃣ Project 何时结束 / 归档

### 可以归档的信号

* 没有活跃 task
* 不再出现在 Daily
* 只剩参考价值

### 归档方式

* 移动到 `90_archived/`
* 保留完整结构
* 不再主动维护

📌 归档 ≠ 删除
它是你的经验库。

---

## 8️⃣ 一句最重要的原则（请记住）

> **Project 不是为了记录
> 而是为了“承载一段持续的注意力”**

如果一个东西已经不占用你的注意力，
它就不该再是 Project。
